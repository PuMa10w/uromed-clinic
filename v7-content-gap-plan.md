# V7 Content Gap Plan (ICD-10 + Premium Clinical Depth)

## 1) Исходная оценка (что уже есть)

- По техническому аудиту покрытие сетки нозологий закрыто:
  - `totalDiseases: 184`
  - `completeMinimum: 184`
  - `highPriorityGaps: 0`
  - `priorityTargetsMissing: 0`
- Вывод: главный дефицит не в отсутствии карточек, а в **клинической глубине**, **актуальности рекомендаций**, **полноте вкладок** и **единообразии структуры**.

## 2) Что считаем “нехваткой информации” в нозологии

Для каждой карточки вводим 4 типа дефицита:

1. `A — Diagnostic Gap`  
Нет четкого алгоритма диагностики (первый шаг → подтверждение → эскалация).

2. `B — Treatment Gap`  
Нет структурированной первой/второй линии, критериев смены тактики и роли хирургии.

3. `C — Safety Gap`  
Нет красных флагов, срочных показаний и маршрутизации (куда/когда направлять).

4. `D — Follow-up Gap`  
Нет наблюдения, сроков контроля, критериев ответа/неэффективности и прогноза.

Дополнительно:
- `E — Evidence Gap`: отсутствуют уровни доказательности/класс рекомендаций.
- `F — Guideline Freshness Gap`: нет года и версии источника (EAU/AUA/EAU Paediatric/RU и т.д.).

## 3) Целевая модель “Clinical Complete v2” для каждой МКБ-10 карточки

Обязательный контент:

- `Overview`: определение + клиническая значимость + 3 ключевых тезиса.
- `Classification`: клинически применимая классификация (не академическая “ради объёма”).
- `Symptoms`: основные и атипичные проявления.
- `Red Flags`: urgent/fatal scenarios, пороги срочного направления.
- `Diagnostics`: пошаговый алгоритм + таблица “что/когда/зачем”.
- `Differential`: минимум 5–10 ключевых дифференциалов (таблица признаков).
- `Imaging/Labs`: нормальные/патологические диапазоны, пороги решений.
- `Treatment`: first-line / second-line / invasive options + switch criteria.
- `Follow-up`: интервалы и контрольные маркеры.
- `Prognosis`: факторы риска неблагоприятного исхода.
- `Clinical Cases`: 1–2 типовых сценария + тактические развилки.
- `Related`: связные нозологии и переходы.

## 4) Приоритизация по волнам (без риска поломок)

### Wave A (критично, 2 недели)

Фокус: состояния с риском потери органа/сепсиса/онко-прогрессии.

- Урология:
  - обструкции (`R33`, `N13.x`, `N35`)
  - инфекции тяжелого течения (`N39.0`, `N41.0`, `N49.2`)
  - камни + колика (`N20.x`, `N23`)
  - онкоядро (`C61`, `C67`, `C64`, `C62`)
- Андрология:
  - `N52`, `F52.2`, `N48.3`, тяжелые формы гипогонадизма (`E29.1`)

Требование выхода волны:
- 100% карточек Wave A имеют закрытые A/B/C/D/E/F gap.

### Wave B (значимо, 2–3 недели)

- LUTS/BPH, CPPS, рецидивирующие ИМП, реконструктивные, функциональные.
- Мужская фертильность: `N46` и подтипы сперматогенеза.

### Wave C (редкие и подрубрики, 3–4 недели)

- Орфанные/редкие формы, педиатрические переходные состояния.
- Уточнение подрубрик МКБ-10 там, где разные тактики ведения.

## 5) Матрица источников (обязательная для каждой карточки)

Для каждой нозологии фиксируем:

- `primary_guideline` (EAU/AUA/другая профильная)
- `secondary_guideline` (локальная/российская адаптация)
- `guideline_year`
- `evidence_notation` (LE/GR или эквивалент)
- `last_review_date`
- `next_review_date`

Приоритет источников:
1. EAU (урология, андрология, онкоурология, инфекции).
2. AUA / ASRM / ISSM (по профилю темы).
3. Российские рекомендации как локальный слой маршрутизации.
4. Доп. международные общества (при отсутствии прямого раздела EAU/AUA).

## 6) Что дополняем по МКБ-10 сверх текущего покрытия

Даже при `covered` в реестре проверяем:

- Есть ли клинически значимые подтипы с иной тактикой.
- Есть ли недостающие “cross-link” карточки для частых коморбидных сценариев.
- Есть ли единая карта переходов:
  - нозология → дифференциал
  - нозология → препарат/группа
  - нозология → калькулятор/опросник

Формат фиксации:
- `icd10-gap-register-v5.csv`:
  - `domain`
  - `icd`
  - `diseaseId`
  - `gapType`
  - `priorityWave`
  - `owner`
  - `status`
  - `sourceNeeded`
  - `targetDate`

## 7) Фарм-раздел: обязательное наполнение

Цель: удобный поиск по препаратам и клиническому эффекту.

Для каждой drug-card:

- INN + торговые группы (без маркетингового смещения).
- Показания в урологии/андрологии.
- Положительная фармакодинамика (что улучшает клинически).
- Отрицательная фармакодинамика/риски (QT, нефро-/гепато-, сексуальная функция, задержка мочи и др.).
- Противопоказания.
- Важные взаимодействия.
- Мониторинг и контрольные точки.
- Связанные нозологии и этапы терапии.

## 8) QA-пайплайн контента и интерфейса (iPhone-first)

Перед каждым релизом волны:

1. Контент QA
- Нет пустых вкладок.
- Нет “заглушек” без клинической ценности.
- Все ключевые решения имеют источник и дату.

2. Технический QA
- `npm run build`
- существующие тесты
- линк-чек межкарточных переходов

3. Визуальный QA (обязательно iPhone)
- Viewports: `320x568`, `375x667`, `390x844`, `414x896`, `428x926`, `667x375`.
- Нет горизонтального переполнения.
- Табы нозологии скроллятся по оси X.
- Блок `Наверх / Закрыть` стабильно фиксирован у нижней safe-area.
- Основной текст ограничен комфортной шириной чтения.

## 10) Definition of Done (строгий)

Карточка считается завершенной только если:

1. Закрыты gap `A/B/C/D`.
2. Есть источник и год (gap `F` закрыт).
3. Есть уровень доказательности или его эквивалент (gap `E` закрыт).
4. Нет визуальных дефектов на iPhone.
5. Карточка связана с релевантными препаратами/калькуляторами/дифференциалами.

## 11) Риски и контроль

- Риск: “формально заполнено, клинически бесполезно”.  
  Контроль: чек-лист практической применимости (можно ли принять решение у постели пациента).

- Риск: устаревание рекомендаций.  
  Контроль: обязательные поля `guideline_year` и `next_review_date`.

- Риск: поломка интерфейса при росте контента.  
  Контроль: iPhone visual smoke matrix + табличные компоненты с адаптивом по умолчанию.
