# Мега подробный план оптимизации сайта UroMed для iPhone

## Введение и цели

Данный план представляет собой комплексный чек-лист и руководство по улучшению сайта UroMed для обеспечения максимально удобного использования на iPhone. Основные цели оптимизации включают: визуальную адаптацию под различные модели iPhone (от iPhone SE до iPhone 14 Pro Max), предотвращение перекрытия иконок и текстовых элементов, обеспечение полноэкранного отображения контента, плавного скроллинга до самого низа страницы без обрезания контента, устранение визуальных ошибок на карточках и разделах, а также достижение интуитивно понятного интерфейса без необходимости объяснять пользователю как что работает. Перед финальным деплоем необходимо провести полный аудит всех моделей данных, компонентов и стилей для выявления и устранения потенциальных ошибок.

## Этап 2: Оптимизация сетки и компоновки

### 2.1 Проблемы с перекрытием иконок

Одной из главных проблем является перекрытие иконок в карточках разделов и кнопках. Для решения этой проблемы необходимо предпринять след��ющие действия:

Во-первых, провести аудит всех кнопок и карточек на предмет минимальной ширины. Согласно рекомендациям Apple для iOS, минимальный размер touch target должен составлять 44x44 пикселя. Необходимо проверить все файлы стилей и убедиться, что ни одна кнопка или интерактивный элемент не имеют размер меньше этого порога. Особое внимание следует уделить кнопкам в навигационной панели (Navbar), карточкам быстрого доступа (quick-btn), кнопкам подразделов (sub-btn) и кнопкам действий в модальных окнах.

Во-вторых, оптимизировать flexbox-компоновку иконок и текста. Во многих компонентах иконки расположены рядом с текстом внутри flex-контейнеров. Необходимо проверить свойство flex-wrap и убедиться, что при недостатке места текст корректно переносится на новую строку вместо перекрытия с иконкой. Также следует проверить свойство flex-shrink и убедиться, что элементы не сжимаются меньше своего минимального размера.

В-третьих, добавить явные отступы (gap) для всех flex и grid контейнеров. Расстояние между элементами должно быть достаточным для четкого визуального разделения. Рекомендуемое минимальное расстояние составляет 8-12 пикселей для мобильных устройств.

### 2.2 Оптимизация ширины для полноэкранного отображения

Для обеспечения адаптации ширины по всему экрану iPhone необходимо рассмотреть следующие аспекты:

Базовые значения ширины контента должны использовать процентные значения или clamp()-функцию вместо фиксированных пикселей. Основной контейнер контента должен иметь max-width: 100% с учетом padding по бокам (min-padding: 16px для iPhone). На больших экранах (iPad, десктоп) можно ограничить max-width до 1200px для читаемости.

Для секций и карточек следует использовать grid с auto-fit и minmax() для автоматической адаптации количества колонок. Рекомендуется использовать формулу: grid-template-columns: repeat(auto-fit, minmax(150px, 1fr)) для карточек и grid-template-columns: 1fr для секций на мобильных устройствах.

Медиа-запросы должны учитывать все популярные размеры iPhone: iPhone SE (320x568), iPhone 12/13/14 (390x844), iPhone Plus (414x896), iPhone Pro Max (430x932). Текущие медиа-запросы в проекте включают breakpoint-ы для 375px, 430px, 667px, 768px и 900px, что покрывает основные модели.

### 2.3 Настройка Safe Area для iPhone X и новее

Текущий проект уже содержит базовые CSS-переменные для Safe Area, но они требуют расширенного применения:

Переменные --sat (safe-area-inset-top), --sar (safe-area-inset-right), --sab (safe-area-inset-bottom), --sal (safe-area-inset-left) должны применяться ко всем fixed и sticky элементам. Особенно важно для навбара (должен учитывать top inset для iPhone с челкой), модальные окна (должны учитывать все insets), нижняя навигация или кнопки (должны учитывать bottom inset для iPhone без кнопки Home).

Дополнительно следует добавить padding-bottom для контента, чтобы при скроллинге до самого низа контент не скрывался за границами экрана. Значение должно динамически рассчитываться с учетом safe-area-inset-bottom.

## Этап 3: Оптимизация скроллинга

### 3.1 Проблема скроллинга до конца

Частая проблема на мобильных сайтах — контент обрезается внизу экрана, и пользователь не может доскроллить до самого конца. Причин может быть несколько:

Во-первых, фиксированные элементы (fixed header, footer, navbar) могут перекрывать контент. Решение: добавить padding-bottom к основному контейнеру контента, равную высоте фиксированных элементов, или использовать sticky вместо fixed где возможно.

Во- Secondly, неправильный расчет высоты viewport. Решение: использовать 100dvh (dynamic viewport height) вместо 100vh для более точного отображения на мобильных устройствах с изменяемой высотой адресной строки браузера.

В-третьих, overflow: hidden на неправильных элементах. Решение: убедиться, что html имеет overflow-y: auto, а body имеет правильный overflow и min-height.

### 3.2 Обеспечение плавного скроллинга

Для максимально плавного скроллинга на iPhone необходимо применить следующие настройки:

webkit-overflow-scrolling: touch следует добавить для всех прокручиваемых контейнеров. Это обеспечивает плавный инерционный скроллинг на iOS.

overflow-y: scroll с -webkit-overflow-scrolling: touch рекомендуется для списков и секций с большим количеством контента.

 overscroll-behavior-y: contain предотвращает нежелательные эффекты при достижении границ скролла ( например, "bounce" эффект, который может мешать навигации).

## Этап 4: Валидация моделей данных

### 4.1 Проверка структуры данных

Перед деплоем необходимо проверить все data files в src/data/ на предмет структурных ошибок. Каждый файл данных должен соответствовать определенной схеме. Основные проверки включают:

Проверка наличия обязательных полей: id (уникальный идентификатор), title (название заболевания), section (раздел), subsection (подраздел). Для некоторых заболеваний также обязательны: icdCode (код МКБ-10), synonyms (массив а��ьтернативных названий).

Проверка типов данных: все поля должны иметь правильные типы (string для текстовых полей, array для массивов, object для вложенных объектов).

Проверка на null/undefined: критические поля не должны иметь null или undefined значений.

Проверка ссылочной целостности: все section и subsection значения должны существовать в sectionData.

В проекте присутствуют более 100 data files, включая основные: urologyData.js, sectionData.js, иrologyData.js, pediatricData.js, а также специализированные по заболеваниям: bphData.js, prostatitisData.js, cystitisData.js, urolithiasisData.js, и многие другие.

Для каждого файла рекомендуется выполнить проверку соответствия схеме и убедиться, что все ссылки на разделы корректны.

## Этап 5: Оптимизация визуальных компонентов

### 5.1 Карточки разделов (Section Cards)

Карточки разделов должны быть оптимизированы для iPhone по следующим параметрам:

Размер: минимальная ширина карточки должна быть не менее 150px для корректного отображения текста. На iPhone SE (320px) карточки должны занимать всю ширину с отступами. На iPhone 12+ (390px) можно отображать 2 карточки в ряд.

Внутренние отступы: padding не менее 12px со всех сторон. Это обеспечивает достаточное пространство для контента внутри карточки.

Иконка и текст: иконка должна быть не менее 24px, текст должен помещаться без перекрытия. Рекомендуется использовать flex-wrap для переноса текста.

### 5.2 Карточки заболеваний (Disease Cards)

Карточки заболеваний в списках требуют особого внимания:

Виртуализация: при использовании react-window (как в VirtualizedDiseaseGrid.js) необходимо убедиться, что высота строки достаточна для контента. overscanRowCount должен быть не менее 3-5 для плавного скроллинга.

Минимальная высота строки: не менее 60px для отображения названия и мета-информации.

Touch target: вся строка должна быть кликабельна, не только текст.

### 5.3 Модальные окна

Модальные окна должны корректно работать на iPhone:

Позиционирование: модальное окно должно учитывать Safe Area сверху (челка) и снизу ( area home indicator).

Скроллинг контента: контент модального окна должен скроллиться независимо от фона. Max-height модального окна должен учитывать viewport и Safe Area.

Закрытие: кнопка закрытия должна быть достаточно большой (минимум 44x44px) и находиться в доступном месте.
